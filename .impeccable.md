## Design Context

### Users

Sendero is used by travel operators, agencies, corporate travel teams, individual travelers, and other AI agents. Humans enter through WhatsApp, web, Slack, Teams, support docs, or a protected workspace. LLMs enter through MCP and llms.txt. The user is usually trying to resolve travel work quickly: search, quote, hold, book, change, cancel, pay, reconcile, or understand whether an agent can act safely.

### Brand Personality

Sendero should feel precise, calm, and agent-native. It is not a SaaS dashboard selling seats. It is a travel operations agent that does real work across channels. The interface should project confidence, restraint, and speed without looking like generic fintech or developer-tool AI chrome.

### Aesthetic Direction

Light-first, refined, operational, and minimal. Use the existing vermilion Sendero accent as a sparse signal, not a wash. Surfaces should feel like a high-end travel operations terminal: readable, structured, low-noise, and immediately useful. Avoid purple/blue AI gradients, glowing dark panels, oversized decorative cards, vague hero copy, nested cards, and redundant auth/wallet narratives.

### Design Principles

1. Agent first, UI second: every surface should make clear what work the Sendero agent can do.
2. One identity story: Clerk manages auth, then protected routes handle tenant, policy, channel, wallet, and billing setup.
3. Public pages must teach quickly: what Sendero is, who it serves, what actions are metered, and how agents discover it.
4. Protected pages should be dense but calm: fast scanning, clear status, no marketing composition.
5. Every AI-agent route must be public, legible, and agent-safe: llms.txt should never be hidden behind sign-in.
